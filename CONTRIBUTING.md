# Contributing to Chronicle / Legacy Protocol

## Purpose

Chronicle / Legacy Protocol is a documentation-first research project. Contributions should help clarify the concept, architecture, risks, research questions, and long-term direction of a Protocol Memory Layer for decentralized ecosystems.

This repository is not currently an implementation project. Contributions should focus on research, documentation, conceptual architecture, governance memory, decision records, and protocol design analysis.

## Contribution Principles

Contributions should follow these principles:

| Principle | Meaning |
|---|---|
| Research before implementation | Define concepts, risks, and assumptions before proposing code |
| Evidence before claims | Support important claims with sources, examples, or clearly stated assumptions |
| Context over promotion | Explain trade-offs and limitations instead of using marketing language |
| Memory before rewards | Focus on preserving contribution and decision context, not farming incentives |
| Reviewability | Write in a way that others can inspect, question, and improve |
| Neutrality | Treat Chronicle / Legacy Protocol as a research subject, not a finished product |
| Safety | Avoid exposing private information, sensitive operational data, or unsupported claims |

## Accepted Contribution Types

Useful contributions may include:

- research notes;
- architecture explanations;
- governance memory scenarios;
- decision records;
- contribution verification models;
- attestation and dispute process analysis;
- knowledge inheritance models;
- AI mentor safeguards;
- archive and preservation risks;
- privacy and disclosure considerations;
- diagrams and conceptual flows;
- issue reports about unclear wording;
- source and reference suggestions;
- improvements to structure, navigation, and terminology.

## Research Note Guidelines

A research note should usually include:

- a clear title;
- the problem being examined;
- relevant context;
- proposed framing or analysis;
- risks and limitations;
- open questions;
- links to related documents;
- a concise conclusion.

Avoid presenting speculative ideas as already implemented or validated.

## Decision Record Guidelines

Decision records should follow the structure in:

- [Decision Record Template](./docs/Decision_Record_Template.md)
- [Sample Decision Record](./research/sample-decision-record.md)

A decision record should preserve context, not force agreement. It should include alternatives considered, evidence, contributor input, risks, disputed points, and later review space.

Decision records should clearly state whether they represent:

- a project-level research decision;
- a hypothetical example;
- a community discussion summary;
- a formal governance process;
- a retrospective analysis.

Do not imply formal governance authority where none exists.

## Evidence and Sources

When adding claims, prefer evidence such as:

- repository links;
- public discussions;
- governance proposals;
- documentation pages;
- academic papers;
- technical specifications;
- archived references;
- clearly identified assumptions.

If evidence is incomplete, state the limitation directly.

## Tone and Style

Use a tone that is:

- professional;
- research-oriented;
- neutral;
- precise;
- readable;
- transparent about uncertainty.

Avoid:

- hype language;
- investment language;
- guaranteed outcome claims;
- reward farming framing;
- exaggerated contributor status;
- official-sounding claims without authority;
- unsupported mainnet, token, or governance assertions.

## What Not to Add

Do not add:

- smart contract code;
- token mechanics presented as final;
- production deployment claims;
- private keys, logs, credentials, or sensitive operational data;
- scraped private community content;
- copied material without attribution;
- promotional announcements;
- unverifiable claims of official status;
- content designed primarily to complete external reward tasks.

This project may later explore specifications or prototypes, but the current stage is conceptual and architectural research.

## Suggested Review Checklist

Before submitting a contribution, check:

- Does it improve conceptual clarity?
- Is the language neutral and research-oriented?
- Are assumptions clearly marked?
- Are important claims supported or limited?
- Does it avoid hype and reward-farming language?
- Does it avoid official claims unless documented?
- Does it fit the existing repository structure?
- Does it preserve uncertainty where needed?

## File Placement

Suggested locations:

| Contribution Type | Recommended Location |
|---|---|
| Core concept note | `docs/` |
| Governance or decision memory material | `docs/` or `research/` |
| Sample decision record | `research/` |
| Ecosystem scenario | `research/` |
| Manifesto or vision text | `manifesto/` |
| Repository navigation updates | `README.md` |

## Review Process

At this stage, review is informal. Contributions should be evaluated for clarity, neutrality, structure, evidence quality, and alignment with the project's research direction.

Future versions may define a more formal review and attestation process.

## Status

These contribution guidelines are part of the concept and architecture research stage. They may evolve as the project becomes more structured.
