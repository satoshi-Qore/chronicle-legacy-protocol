# 02 - Contributor Reputation

## Overview

Contributor reputation is the interpretive layer built from verified contribution history. It should not be treated as a transferable token, market asset, or universal score. Reputation should remain contextual, evidence-aware, and tied to contribution history.

## Contextual Reputation

A contributor may have different forms of reputation across different domains:

- Protocol research
- Documentation
- Code contribution
- Infrastructure operation
- Governance participation
- Education and onboarding
- Security review
- Translation and localization
- Community coordination

A single score cannot accurately represent all of these categories.

## Non-Transferability

Reputation should not be bought, sold, or transferred. If contributor reputation becomes transferable, it stops representing lived contribution and becomes a market object.

Possible design directions include:

- Non-transferable contribution credentials
- Pseudonymous contributor profiles
- Attestation-linked reputation records
- Category-specific reputation views
- Lifetime and recent contribution separation

## Risks

| Risk | Description |
|---|---|
| Score reductionism | Complex human contribution becomes flattened into a number |
| Reputation farming | Contributors optimize for visible metrics instead of useful work |
| Reviewer centralization | A small group controls recognition |
| Privacy exposure | Contributors reveal more personal history than intended |
| Governance capture | Reputation becomes a political control mechanism |

## Design Position

Reputation should help ecosystems remember useful work. It should not replace judgment, governance deliberation, or evidence review.
