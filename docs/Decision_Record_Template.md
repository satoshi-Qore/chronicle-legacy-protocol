# Decision Record Template

## Purpose

This template provides a structured format for recording governance decisions, ecosystem design choices, protocol-level research conclusions, and other historically significant decisions within Chronicle / Legacy Protocol.

The goal is not to create bureaucracy. The goal is to preserve decision context so future contributors can understand what was decided, why it was decided, which alternatives were considered, and what evidence shaped the outcome.

Decision records should remain factual, source-linked, and open to later review.

## Template Format

### Decision Record: [Short Decision Title]

### 1. Metadata

| Field | Entry |
|---|---|
| Decision ID | DR-[YYYY]-[NUMBER] |
| Title | [Short title of the decision] |
| Date | [YYYY-MM-DD] |
| Ecosystem Phase | [Concept / Testnet / Pre-mainnet / Mainnet / Post-mainnet / Other] |
| Status | [Proposed / Accepted / Rejected / Superseded / Under Review / Archived] |
| Authors | [Names, handles, or pseudonymous contributor references] |
| Reviewers | [Maintainers, governance participants, domain reviewers, or community reviewers] |
| Related Documents | [Links to related proposals, issues, discussions, research notes, or records] |

### 2. Problem Statement

Describe the problem, decision point, or governance question being addressed.

A good problem statement should explain:

- what issue required a decision;
- why the issue mattered;
- who or what was affected;
- what uncertainty existed at the time;
- what would likely happen if no decision were made.

### 3. Context and Background

Provide historical and technical context.

Include relevant information such as:

- prior decisions;
- earlier proposals;
- governance discussions;
- technical constraints;
- ecosystem conditions;
- community concerns;
- implementation dependencies;
- known limitations.

### 4. Options Considered

List the main alternatives that were discussed.

| Option | Description | Potential Benefits | Risks or Limitations |
|---|---|---|---|
| Option A | [Description] | [Benefits] | [Risks] |
| Option B | [Description] | [Benefits] | [Risks] |
| Option C | [Description] | [Benefits] | [Risks] |

This section should avoid presenting the chosen option as obvious. It should preserve trade-offs fairly.

### 5. Evidence and References

List the evidence used to support the decision.

| Evidence Type | Source | Relevance | Limitations |
|---|---|---|---|
| Proposal | [Link] | [Why it matters] | [Known limitations] |
| Discussion | [Link] | [Why it matters] | [Known limitations] |
| Technical Document | [Link] | [Why it matters] | [Known limitations] |
| Audit or Review | [Link] | [Why it matters] | [Known limitations] |
| Community Feedback | [Link] | [Why it matters] | [Known limitations] |

Evidence quality should be stated clearly. Weak, incomplete, or disputed evidence should not be hidden.

### 6. Contributor Input

Record meaningful contributor participation without overstating influence.

| Contributor | Role | Contribution Type | Evidence Link | Notes |
|---|---|---|---|---|
| [Name/Handle] | [Researcher / Operator / Developer / Community / Reviewer / Other] | [Summary] | [Link] | [Scope and limitations] |

Contributor input may include research, review, documentation, translation, operator feedback, issue analysis, governance discussion, or educational work.

### 7. Decision Outcome

State the final decision clearly.

Include:

- selected option;
- decision rationale;
- expected benefits;
- accepted risks;
- unresolved issues;
- implementation assumptions;
- conditions that may trigger later review.

### 8. Governance Result

If the decision involved formal or informal governance, record the result.

| Field | Entry |
|---|---|
| Governance Process | [Vote / Maintainer decision / Community review / Research conclusion / Other] |
| Voting Result | [If applicable] |
| Quorum / Participation | [If applicable] |
| Timeline | [Start and end dates] |
| Final Status | [Accepted / Rejected / Deferred / Revised] |

If no formal vote occurred, state that clearly.

### 9. Risks and Trade-Offs

Document known risks and trade-offs.

| Risk | Description | Mitigation | Open Concern |
|---|---|---|---|
| [Risk 1] | [Description] | [Mitigation] | [Remaining concern] |
| [Risk 2] | [Description] | [Mitigation] | [Remaining concern] |
| [Risk 3] | [Description] | [Mitigation] | [Remaining concern] |

This section should preserve uncertainty rather than remove it.

### 10. Disputed Points

Record disagreements or unresolved interpretations.

| Disputed Point | Position A | Position B | Current Treatment |
|---|---|---|---|
| [Issue] | [Viewpoint] | [Alternative viewpoint] | [How the record handles it] |

Decision records should not erase disagreement. They should make disagreement understandable.

### 11. Implementation or Follow-Up Plan

If implementation is required, describe the expected next steps.

| Task | Owner or Responsible Group | Status | Evidence Link |
|---|---|---|---|
| [Task] | [Owner] | [Pending / In Progress / Completed / Blocked] | [Link] |

If the record is conceptual and does not require implementation, state that clearly.

### 12. Retrospective Review

This section may be completed later.

| Review Date | Observation | Evidence | Revision Needed? |
|---|---|---|---|
| [YYYY-MM-DD] | [What was learned] | [Link] | [Yes / No / Under Review] |

Retrospective review helps prevent governance memory from becoming static or misleading.

### 13. Revision History

| Version | Date | Change | Contributor |
|---|---|---|---|
| v0.1 | [YYYY-MM-DD] | Initial draft | [Contributor] |
| v0.2 | [YYYY-MM-DD] | [Change summary] | [Contributor] |

## Notes on Use

Decision records should:

- preserve context rather than promote a single narrative;
- cite sources where possible;
- distinguish facts from interpretations;
- record uncertainty and limitations;
- avoid exaggerated contributor claims;
- remain open to correction and later review;
- support governance memory without creating permanent authority.

This template is a research-stage format and may evolve as Chronicle / Legacy Protocol develops.
